# AI 工作台能力决策清单

更新时间：2026-07-13

本文用于记录 Pier 的能力边界和决策状态。它不再用理论评分自动生成实施路线，也不把外部产品已有的能力直接视为 Pier 的架构缺口。

## 1. 产品边界与判断标准

Pier 是本地 AI 开发工作台，核心范围是：

- 稳定终端；
- 多面板工作区布局；
- Git 变更与文件查看；
- 多 Agent 活动状态；
- 可信的内置插件和官方受管理插件。

Pier 不建设任务生命周期、SQLite 任务台账、看板、自动调度、通用自动化平台、远程执行平台或第三方插件市场。

### 1.1 能力进入路线图的硬门槛

一项新能力只有满足以下条件，才可以进入实施路线：

1. 存在明确、可复现的用户痛点，或者至少有两个真实消费者需要共同能力。
2. 当前能力已经正确工作时，不得仅以“未来可能需要”为理由增加数据域、状态机或持久化模型。
3. 第三方插件能力只有在产品明确决定开放第三方后，才启动安全运行时设计。
4. 某个插件的发展方向不会自动转化为宿主公共能力；应先由插件在自己的边界内验证需求。
5. 已完成能力只保留维护责任，不继续制造“补齐闭环”任务。

### 1.2 决策状态


| 状态       | 含义                                 |
| -------- | ---------------------------------- |
| 已完成，持续维护 | 已满足当前产品目标；后续只处理故障、兼容性、回归和有证据的性能问题。 |
| 需求触发型候选  | 当前没有实施任务；达到本文写明的启动条件后重新评估。         |
| 明确不做     | 与当前产品边界冲突，或已有更合适的外部工具承担。           |
| 未来准入门槛   | 不是当前缺口；只有产品边界发生明确变化时才必须完成。         |

## 3. 需求触发型候选项及启动条件

候选项不是待办事项。未达到启动条件时，不分配实施任务，也不预先创建数据表、持久化契约或公共 API。


| 候选项              | 启动条件                                  | 约束与停止条件                                                                                                                                                                                |
| ---------------- | ------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 跨会话证据            | 出现可复现的跨会话验收需求，或至少两个真实消费者需要共享证据。       | 不绑定任务域，不顺带建设宿主 Transcript；先定义最小证据对象和删除策略。                                                                                                                                              |

| 继续上下文辅助          | 提供方原生继续能力无法覆盖重复出现的交接问题。               | 先验证提供方能力；不复制完整会话或建设统一 transcript。                                                                                                                                                      |
| 轻量引用             | 用户需要在面板或会话间传递标题、路径、网址等引用，并出现稳定工作流。    | 只设计轻量引用，不引入任务状态、负责人、排期或看板。Agent↔Canvas 绑定的持久边由插件图模型持有；宿主 Index 只提供运行时 facade，可选 join 见 [Agent Runtime Index 与注意力通知](./2026-07-15-agent-runtime-index-and-attention-design.md)。 |

## 4. 明确不做的能力

| 能力                         | 不做原因或替代方案                                                        |

| 重型权限策略引擎                   | 当前可信代码范围使用 capability、manifest 校验和按客户端类型授权；第三方安全需要主体隔离，不是增加规则数量。 |

| 自动错误诊断平台                   | Pier 提供现场和明确反馈，模型负责解释和建议。                                        |
| 远程执行、SSH、runner            | Pier 的定位是本地工作台。                                                  |
| 跨设备、移动端和多人协作               | 不属于当前产品目标。                                                       |
| 专业 Office、PDF 制作和多媒体生成     | 使用专业外部工具；Pier 只覆盖工程文件查看。                                         |
| 多 Agent 自动编排、任务 DAG 和调度    | 与明确的产品边界冲突。                                                      |
| 个人信息工作区                    | 不属于开发工作台范围。                                                      |
| thread automation 或通用自动化平台 | 交给 Codex App 或专门自动化系统。                                           |


### 4.1 第三方插件开放前的强制准入条件

当前只允许内置插件和官方受管理插件。external main 是可信代码，以普通 Node ESM 运行；`pluginId`、capability 断言、manifest 校验和包扫描是工程纪律，不构成恶意代码隔离。

未来若产品决定开放第三方插件，至少必须先完成：

1. 独立进程或独立 realm 隔离；
2. 每插件独立主体身份；
3. main 侧按插件主体授权；
4. 最小权限 host API；
5. 权限撤销和运行中止机制；
6. 可查询的审计记录；
7. 供应链签名、来源约束、版本回滚和紧急撤回。

这些条件是未来准入门槛，不是当前路线图。生产环境继续忽略任意本地、git、registry、自建 marketplace、`PIER_OFFICIAL_PLUGIN_INDEX_URL` 和历史 dev override 路径。

原任务分组不再作为主动路线图：

- Agent 审计的已完成项保留证据；状态识别和适配器检查只在新增提供方或出现误报时执行，诊断界面不进入路线图。
- Evidence 相关工作全部改为需求触发型候选。
- Profile 相关工作全部移出主动路线；项目或工作树绑定只在真实多项目配置痛点出现时恢复评估。
- 第三方隔离和供应链要求改为开放第三方前的强制准入门槛；官方插件的审计或后台能力由实际插件需求触发。
- Trigger / Action 共同协议只有在至少两个真实插件需要时才重新设计。
- 现有受管理插件测试继续随具体改动补充；性能与故障专项验证由事故或测量数据驱动，不设置无边界长期任务。

## 6. 当前真实行动项

当前只保留以下行动：

1. 维护现有核心能力及其测试。
2. 新增 Agent 或提供方时，补对应适配器测试。
3. 官方插件安装、更新、恢复或回滚链路发生变更时，补对应故障测试。
4. 未满足需求触发条件时，不启动新的平台数据域。

`profile:read`、`evidence:write` 等预留 capability 当前没有真实消费者。本文不把它们描述成已交付能力，也不在文档任务中删除运行时代码；是否清理由独立代码任务评估。
