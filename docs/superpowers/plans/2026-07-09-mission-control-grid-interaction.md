# 指挥中心网格交互实施计划

> 面向执行者：本计划用于把指挥中心从“默认自动重排展示”修正为“成熟网格交互优先”。实现时按测试先行推进，每完成一步同步勾选。

## 目标和完成标准

默认状态下，指挥中心必须尊重用户持久化的 12 列布局，并在未锁定时恢复 `react-grid-layout` 的拖拽、调整尺寸、碰撞处理和压实能力。自动整理只作为显式菜单动作，不在渲染时偷偷改写展示结果。

完成标准：

- 未锁定布局：有可识别的拖拽提示和右下角调整尺寸手柄。
- 锁定布局：不可添加、不可删除、不可复制、不可拖拽、不可调整尺寸。
- 默认渲染：12 列宽容器下保留持久化 `x/y/w/h`，不自动扩宽卡片。
- 显式整理：用户选择“整理布局”后，才运行自动布局求解器并写回新的 12 列基准布局。
- 窄容器：只派生显示布局，纯挂载回声不写回参数；真实编辑仍通过派生差分合并回 12 列基准。

## 所有权划分

- 数据：`MissionControlPanelParams.widgets` 仍是唯一持久化来源，按 12 列基准保存。
- 策略：`mission-control-auto-layout.ts` 只负责显式整理算法，不参与默认渲染接管。
- 执行：`react-grid-layout` 负责拖拽、调整尺寸、碰撞和压实。
- UI：`mission-control-panel.tsx` 负责选择基准布局或窄容器派生布局，并暴露“整理布局”菜单动作。
- 状态：`use-mission-control-panel-state.ts` 负责所有参数写回和显式整理写回。
- 测试：组件测试锁定默认可编辑、锁定禁用、显式整理和派生模式不误写。

## 数据流

默认 12 列：

`params.widgets` -> `entryToLayoutItem` -> `GridLayout.layout` -> 用户拖拽/调整尺寸 -> `handleLayoutChange` -> `api.updateParameters({ widgets })`

窄容器：

`params.widgets` -> `entryToLayoutItem` -> `deriveLayout(cols)` -> `GridLayout.layout` -> 纯挂载回声丢弃；真实编辑经 `applyDerivedLayoutChange` 合并回 12 列基准。

显式整理：

右键菜单“整理布局” -> `deriveOptimalAutoLayout(..., 12)` -> `layoutToEntries` -> `mergeGeometry` -> `api.updateParameters({ widgets })`

## 禁止的反模式

- 渲染期间无条件运行自动布局并覆盖用户布局。
- 把窄容器派生坐标直接写回 12 列基准。
- 为了避免复杂交互而关闭拖拽和调整尺寸。
- 重新暴露手动尺寸预设菜单，让用户在卡片菜单里选择裸网格尺寸。
- 绕过组件声明的最小/最大尺寸约束。

## 验收矩阵

| 需求 | 证据 |
| --- | --- |
| 默认未锁定可编辑 | 组件测试断言拖拽提示和调整尺寸手柄存在 |
| 锁定后不可编辑 | 组件测试断言添加入口、编辑菜单和调整尺寸手柄隐藏 |
| 默认尊重持久化布局 | 12 列组件测试断言两个 4 列卡片仍是 4 列宽 |
| 显式整理才自动扩宽 | 组件测试点击“整理布局”后断言写回 6+6 布局 |
| 窄容器不误写 | 现有派生模式守卫测试保持通过 |
